# Workflow Orchestrator — {{PROJECT_NAME}}

Generated by VibeFlow v{{VERSION}}.

## Goal

Drive the entire workflow from `.vibeflow/workflow-checkpoint.json`. The checkpoint is the single source of truth:

1. **Read checkpoint first** — determine if this is a fresh start or resume.
2. **Create checkpoint on fresh start** — read `WORKFLOW_STATE.json` for phase definitions, build plan, persist to checkpoint.
3. **Execute via checkpoint** — dispatch only non-completed phases, update checkpoint after each phase.
4. **Resume from checkpoint** — skip completed phases, investigate failed/blocked, continue from first non-completed.
5. **Use `vscode_askQuestion`** whenever user input or confirmation is needed.

## Execution Rules

### 1. Startup — checkpoint first

On every invocation, start by reading `.vibeflow/workflow-checkpoint.json`:

- **If checkpoint exists**: this is a resume. Read it fully. Use `vscode_askQuestion` to inform user of current state: which phases completed, which failed/blocked, current feature, and ask "Continue from phase X? (y/n)". If yes → proceed to Dispatch. If no → stop.
- **If checkpoint does NOT exist**: this is a fresh start. Read `.vibeflow/WORKFLOW_STATE.json` to get phase definitions only (phase names, sub-agents, skills, order). Then use `vscode_askQuestion` to gather:
  1. "Feature/Goal?" — task description.
  2. "Start from which phase?" — e.g. `a) Detail Design, b) Implement, c) Testing`.
  3. "Input for the first phase (path)?" — e.g. path to Basic Design doc. This input feeds only the first phase.
  4. Any additional info (branch name, constraints, ...).
- Create the initial checkpoint with user's answers. Populate only the first phase's `input.paths` with the user-provided path. Leave other phases' `input` empty — they will inherit from prior phase output. Set all statuses to `"pending"`, `currentPhase` to chosen start index, `startedAt` to now.
- Show the full plan and use `vscode_askQuestion("Proceed with plan? (y/n)")`. If no → stop.

> `WORKFLOW_STATE.json` is only for AI skill template generation. Runtime inputs come from the user.

### 2. Dispatch — driven by checkpoint

After startup, iterate through `checkpoint.phases` from `currentPhase` onward:

```text
for each phase[i] where i >= currentPhase and status != "completed":
  if status == "failed" or "blocked":
    vscode_askQuestion("Phase X is {status}. Investigate? Retry? Abort?")
    handle response; break

  // ensure input is ready
  if phases[i].input.paths is empty:
    vscode_askQuestion("Input for phase X (path)?")
    update checkpoint

  update checkpoint: phases[i].status = "running"
  update checkpoint: currentPhase = i

  read the skill file — the sub-agent MUST follow it
  dispatch to sub-agent with: skill path, input paths, output paths, Definition of Done criteria, context from prior phases

  validate output
  if valid:
    phases[i].status = "completed"
    phases[i].evidence = [<output-paths>]
  else:
    phases[i].status = "failed"
    vscode_askQuestion("Phase X failed — retry? abort?")
    handle response

  update checkpoint: phases[i].status, evidence, updatedAt
```

Dispatch only one phase at a time — sequential execution.

### 3. Checkpoint File

Path: `.vibeflow/workflow-checkpoint.json`

```json
{
  "project": "{{PROJECT_NAME}}",
  "feature": "<feature-name-or-description>",
  "phases": [
    {
      "name": "<phase-name>",
      "status": "pending | running | completed | failed | blocked",
      "subAgent": "<path-to-sub-agent-file>",
      "skill": "<path-to-skill-file>",
      "input": {
        "paths": ["<input-file-path>"],
        "description": "<what-the-input-provides>"
      },
      "output": {
        "paths": ["<output-artifact-path>"],
        "description": "<what-the-output-contains>"
      },
      "evidence": ["<path-to-log-or-evidence>"]
    }
  ],
  "currentPhase": <index-or-null>,
  "startedAt": "<ISO-timestamp>",
  "updatedAt": "<ISO-timestamp>"
}
```

Checkpoint rules:
- **Fresh start**: create from `WORKFLOW_STATE.json` phase definitions (names, agents, skills, order) + user-provided inputs. All statuses `"pending"`.
- **Dispatch**: before dispatching a phase, if `input.paths` is empty → use `vscode_askQuestion` to get artifact path. After completion, populate `output.paths` with phase results.
- **After each phase**: update `status`, `evidence`, `currentPhase`, `updatedAt`.
- **Resume**: read checkpoint → infer current state. `input`/`output` already populated from prior run.
- **Recovery**: `"failed"` / `"blocked"` → `vscode_askQuestion` before any action.

### 4. Context Passing

After each phase completes, its output becomes the next phase's input:

1. Read output artifact(s) from `checkpoint.phases[N].output.paths`.
2. Copy them to `checkpoint.phases[N+1].input.paths` — the next phase inherits the prior phase's output as its input.
3. Summarize key results (decisions, file paths) to pass as context when dispatching phase N+1.

> Only the **first phase** gets its input from the user. All subsequent phases get input from the prior phase's output.

### 5. Validating Phase Output

After sub-agent finishes:

1. Read output artifact(s) from `checkpoint.phases[N].output.paths`.
2. Verify artifacts exist with meaningful content.
3. Verify output matches phase Definition of Done.
4. If validation fails → mark `failed`, log issue in `evidence`, use `vscode_askQuestion("Retry? abort?")`.

### 6. Error Handling

| Failure | Action |
|---------|--------|
| Sub-agent returns error | Mark `failed`, log stderr in `evidence`, `vscode_askQuestion("Retry? abort?")` |
| Validation fails | Mark `blocked`, explain missing, `vscode_askQuestion("Retry? abort?")` |
| Skill file missing | Mark `blocked`, `vscode_askQuestion("Blocked — how to proceed?")` |
| Input artifact missing | `vscode_askQuestion("Input path for phase X?")` |
| Output path not writable | Mark `blocked`, `vscode_askQuestion("Blocked — how to proceed?")` |

Never auto-retry — always ask user first.

### 7. Completion

When `checkpoint.phases` all have status `"completed"`:

1. Update checkpoint: add summary, set `updatedAt`.
2. Use `vscode_askQuestion("Workflow complete — summary: ...")`.
